## 一、★ 战报数据编码与解码

### 1.1 编码格式（已完全破解）

战报的 `team_battle_data` / `enemy_battle_data` 字段：
```
原始字符串 = base64( [4字节小端header=明文长度] + zlib_compress( JSON文本 ) )
```

**解码步骤**：
```python
import base64, zlib
raw = open('enemy_battle_data.txt').read()        # base64字符串
decoded = base64.b64decode(raw)                     # 二进制
header = int.from_bytes(decoded[:4], 'little')      # 明文长度
plaintext = zlib.decompress(decoded[4:]).decode('utf-8')  # JSON明文
data = json.loads(plaintext)                        # 解析
```

### 1.2 解码后数据结构

```python
{"<team_id>": [
    块0,   # 舰队配置 [[ship_id, blueprint, type, ..., 武器槽, 模块串], ...]
    块1,   # ★HP曲线字符串 "type,t,struct,attacker_struct#..."
    块2    # 舰船详细+攻击事件
]}
```

### 1.3 ★ HP曲线格式（关键修正）

```
"type,time,struct,attacker_struct#type,time,struct,attacker_struct#..."
```

| 列 | 含义 | 说明 |
|---|---|---|
| type | 事件类型 | 5=常规采样, 97=击毁, 99=team信息 |
| time | 时刻(秒) | 相对于begin_time |
| **struct** | **受击方结构值** | 第3列，被攻击方的当前结构 |
| **attacker_struct** | **攻击方总结构值** | 第4列，所有攻击方的当前总结构 |

**注意**：
- 第3列是**被攻击方**（HP曲线所属方）的结构
- 第4列是**攻击方**的总结构（记录反击造成的伤害）
- 之前误以为第4列是"护盾"，实际是攻击方总结构

### 1.4 type=97 击毁事件

```
97, <击毁时刻>, <击毁者ship_uid>, 0
```
- 击毁时刻是**精确的**（非5秒采样，是事件级）
- **后摇 = 战报时长 − 击毁时刻 = 1秒**（实测，非04文档假设的5秒）

## 二、★ 命中率精确分析

### 2.1 混沌级轨道炮 vs CV3000（核心数据）

**配置**：2艘混沌(bp=51201) 打 1艘CV3000(bp=80201)

**参数**：
```
主炮: dph=1200, 抵抗270, perHit=930, CD=8.4(面板加点), 锁定8, 弹药2×次数1=2发/周期
副炮: dph=20, 抵抗270不破防, perHit=2(10%保底), 持续4s打4发, CD=6, 锁定4, 2门
CV3000: 结构370192, 无闪避(dodge=0)
```

**两场交叉验证**：

| | 无加成(1349s) | 有加成+25%(1083s) |
|---|---|---|
| 战报时长 | 1349s | 1083s |
| 击毁时刻 | 1348s | 1082s |
| 后摇 | 1s | 1s |
| 主炮命中 | 394发 | 395发 |
| 主炮命中率p | **61.6%** | **77.1%** |
| base(去加成p÷1.25) | 61.6% | **61.7%** ✓ |
| 副炮伤害(UI) | 3772 | 2912 |

**结论**：
- 主炮base = **61.6%**（面板50-70%，两场一致）
- 加成+25%后 p=77.1%，÷1.25=61.7%——**加法公式全额生效，无衰减**
- 命中发数固定≈394（因靶HP固定，击毁所需命中数固定）

### 2.2 struct伤害分解方法

CV3000结构伤害 = 主炮(930/发) + 副炮保底(2/发)：
```python
for 每段Δstruct:
    n_主 = round(Δ / 930)
    微调n_主使残差 = Δ - n_主×930 ≥ 0 且为2的倍数
    n_副 = 残差 / 2
```
验证：394×930 + 1886×2 = 366420 + 3772 = 370192 = CV3000总结构 ✓

### 2.3 时间参数（实测修正04文档）

| 参数 | 04文档假设 | 实测值 | 来源 |
|---|---|---|---|
| **后摇** | 5s | **1s** | 战报时长−击毁时刻(type=97) |
| **前摇T0** | 锁定=前摇 | **≈7s** | 主副周期数交叉验证 |
| 主炮CD | 8.4s | 8.4s(面板) | 面板加点值 |
| 副炮周期 | — | 10s(持续4+冷却6) | 面板 |
| 首发时刻 | — | T0+锁定=15s | HP曲线首次掉血@t=15 |

**前摇反推方法**：
- 主炮收尾（struct归零）：`击毁时刻 = T0 + 锁定8 + (k-1)×8.4`
- 副炮：`最后发时刻 = T0 + 锁定4 + (k_s-1)×10`
- 主副T0应相同 → 交叉验证锁定唯一解

## 三、★ 命中率加成衰减谜题——最终结论

### 04文档§2的"谜题"
> "+30%命中加成在多舰下只兑现58%"

### 本轮结论：**无衰减，是公式理解的数学必然**

```
命中率公式: p = base × (1 + k)
  base = 基础命中率（如混沌61.6%）
  k    = 命中加成（如+0.25）

命中率提升(绝对pp): Δp = base × k
"兑现率" = Δp / k = base

→ "兑现率58%"不是衰减，是 base≈0.58 的体现
→ 04文档把乘法结构 base×(1+k) 误读为"加成衰减"
```

**验证**：
- 混沌+25%：p从61.6%→77.1%，DPS比1.2456 vs 预测1.25（偏差0.4%）
- 两场base独立反推一致（61.6% vs 61.7%）
- **加法公式在N=2多舰下全额生效**

## 四、★ 数据流总结（客户端 vs 服务器）

```
服务器: 计算整场战斗 → 生成命令序列 → 编码成战报
                                    ↓
客户端: 收到战报 → 解码HP曲线 → 按时间轴播放动画

HP曲线(服务器原始数据):
  第3列 = 受击方结构值（逐5秒采样）
  第4列 = 攻击方总结构值（记录反击伤害）
  type=97 = 精确击毁时刻

客户端不做任何命中/伤害计算（只播放）：
  WeaponAttackCommand的字段(hit_damage_count/miss_damage_count/base_hit_damage)
  全是服务器预先算好的
```

## 五、对引擎实现的指导

### 5.1 确认可落地的参数
```
命中率: base×(1+k-dodge), 夹[10%,95%]  ← 04文档公式，本轮验证成立
伤害: 离散，每命中扣perHit整数倍        ← 第N次确认
实弹: perHit = max(dph-resistance, dph×10%)
能量: perHit = dph×(1-shield%)
后摇: 1s（非5s）
前摇: ≈7s（T0），含锁定+额外延迟
```

### 5.2 引擎校准基准
- HP曲线是服务器原始数据，引擎输出逐5秒HP可直接对比
- 混沌base=61.6%可作为命中率校准锚点
- 命中发数=HP/perHit（固定），命中率只影响时长

### 5.3 待精确的参数
- 前摇T0的精确值（7s是主副交叉估计，5秒采样限制）
- 副炮base（混沌副炮67%，但面板区间未确认）
- 第4列attacker_struct的GCD=5含义（CV3000反击perHit）
