# 蓝图巅峰等级与调校系统机制

> 日期：2026-07-03（初版）/ 2026-07-04（巅峰奖励 + 调校联动补全）
> 状态：巅峰系统已实装（含奖励强化项），调校系统已实装（含强化联动）
> 代码：`packages/engine/src/data/peakLevel.ts`、`tuneSystem.ts`、`blueprintResolver.ts`

## 〇、★架构区分：蓝图数据层 vs 战斗实例数据层

**这是理解整个数值系统的前提。**

```
【蓝图数据层】（静态、设计态、可编辑、source of truth）
  = 蓝图本身定义：模块组合 + 强化加点 + 调校 + 巅峰等级 + 技术值
  = 舰船的"设计图/配方"，战斗不影响它
  = 对应引擎 packages/engine/src/data/* （resolveBlueprint 等读取数据来源）

【战斗实例数据层】（动态、运行态、一次性、derived）
  = 客户端从蓝图聚合出战斗所需的最终数值（结构值/火力/抵抗等）
  = 传给服务器，服务器算战斗，返回结果
  = 战斗中结构值损失针对"实例"，不回写蓝图
  = 每次战斗从蓝图重新聚合一份新实例
  = 对应引擎 packages/engine/src/core/* + model/* （simulator/runtime）
```

**面板结构值是分层聚合后的总值**（战斗实例数据层）：
- 面板显示 = 各来源（模块基础 + 强化 + 调校 + 巅峰 + 版本号）聚合后的数值
- 点击展开 = 显示每个来源的明细贡献（来自巅峰等级X值、来自调校X值、来自版本号X值、来自各强化槽X值...）

**resolveBlueprint 的职责 = 蓝图态→聚合数值的转换器**（模拟客户端聚合逻辑）。
它把蓝图各来源的加成全部聚合，输出面板/战斗所需的总值。引擎不需要"专门走来源"，直接取聚合后的值。

**field[1] 巅峰奖励的结构加成应叠加进总值**（它是合法来源）。bull-panel 旧测试的 45948 基于不完整聚合（漏了 field[1]），已修正为 46669。

## 核心结论（先读这个）

**巅峰系统和调校系统是两个完全独立的系统**，数据各自独立，机制不同：

| 维度 | 巅峰系统 | 调校系统 |
|---|---|---|
| 等级上限 | 20级 | 10级 |
| 解锁方式 | 道具开启 + 任务经验，逐级提升 | 三步解锁链，与巅峰无必然关系 |
| 加成频率 | 每级都有结构加成(field[0])；特定等级解锁奖励强化项(field[1]) | 永久属性加成（解锁时）+ 调校等级提升 |
| 本次状态 | ✅ 已实装（含 field[1] 奖励强化项） | ✅ 已实装（含强化联动 + 效果接入面板） |

## 一、巅峰系统（已实装）

### 1.1 数据来源

`cfg_ship_peak_level.json`，key = `shipId(5) + peakLv(2)`，如 `6030116` = ST59 巅峰16级。
value = `[强化串, 调校解锁串]`：
- 字段[0] 强化串：`"slotId,level;slotId,level;..."` —— **巅峰系统**
- 字段[1] 调校解锁串：`"slotId70,level;..."` —— **调校系统**（本次不实装）

### 1.2 强化串机制（字段0）

格式：`"6030101,16;6030102,1;6030103,2;"`

- 每段 `slotId,level`：slotId（7位）= shipId(5) + slot(2)
- level = 该槽 **optIdx01（主强化项）的等级**，巅峰把上限从普通4抬到20

**槽位语义**（以 ST59/60301 为例）：
| 槽 | 系统 | EFFECT_ID | 效果 | 出现频率 |
|---|---|---|---|---|
| 槽01 (6030101) | 龙骨结构 | 12 | 结构值绝对值加成 | **每级都有**（1→20） |
| 槽02 (6030102) | 引擎出力 | 14 | 常规移速% | 巅峰10级首次 |
| 槽03 (6030103) | 曲率引擎 | 16 | 曲率移速% | 巅峰5级首次 |

### 1.3 ★数值来源（已铁证验证）

查**船级专属 effect 表** `cfg_system_effect[slotId + "01"].EFFECT_PARAM_LEVEL` 第 N 行。

该表本身覆盖 1-30 级。普通玩家受 `ENHANCE_COST` 长度限制只能升到4级，巅峰把可升级上限抬到20，读取表中本就存在的行。

**校验锚点**（实测一致）：

| 舰船 | shipId | 巅峰等级 | 查表 key | EFFECT_ID | 数值 |
|---|---|---|---|---|---|
| ST59 战巡 | 60301 | 16 | 603010101 第16行 | 12 龙骨 | +52680 结构 |
| ST59 战巡 | 60301 | 20 | 603010101 第20行 | 12 龙骨 | +66905 结构 |
| FG300 护卫 | 30101 | 20 | 301010101 第20行 | 12 龙骨 | +3245 结构 |
| 斗牛 驱逐 | 40501 | 5 | 405010101 第5行 | 12 龙骨 | +2745 结构 |

斗牛巅峰5级 = 2745，与之前用经验公式估算的值完全一致（之前标注"≈floor(版本号×0.7×巅峰/5)"是误判，实际是 PARAM_LEVEL 查表）。

### 1.4 实装代码

- `peakLevel.ts`：
  - `parsePeakEnhanceString(str)` 解析强化串
  - `computePeakBonus(store, shipId, peakLevel)` 聚合结构/移速加成
  - `getPeakLevelByExp(store, totalExp)` 按累计经验判定等级
- `blueprintResolver.ts`：
  - `BlueprintOptions.peakLevel`（取代旧 `peakStructureBonus` 绝对值 hack）
  - EFFECT_ID=12 → `peakStructureBonus`（结构绝对值）
  - EFFECT_ID=14 → `speedBonus`（常规移速万分比）
  - EFFECT_ID=16 → `curvatureSpeedBonus`（曲率移速万分比）
- 返回 `ResolvedBlueprint.peakLevel / peakStructureBonus / peakSpeedBonus / peakCurvatureSpeedBonus`

### 1.5 巅峰等级判定（经验阈值）

`cfg_blueprint_peak_level.json`：key=巅峰等级，value=[等级, 累计经验]。
```
巅峰1=10000, 巅峰5=20000, 巅峰10=46000, 巅峰16=90000, 巅峰19=125000, 巅峰20=0(满级)
```
等级靠道具开启 + 任务经验提升（任务系统在服务端，客户端只存任务ID列表于 `cfg_peak_level_auth.json`）。

### 1.6 ★巅峰强化奖励（field[1]，2026-07-04 突破）

`cfg_ship_peak_level[shipId+peakLv]` 是 2 元素数组：
- **field[0]** = `"slotId,level;..."` —— 普通强化项等级给定（slot01/02/03，§1.2 已述）
- **field[1]** = `"enhanceId,level;..."` —— **★巅峰专属强化项等级给定**（本次新增实装）

field[1] 解析链（以斗牛40501巅峰5 `405010270,2` 为锚点验证）：
1. enhanceId=`405010270`（9位 = slotId 4050102 + optIdx 70）
2. 查 `cfg_system_enhance[405010270].SYSTEM_EFFECT_PREFIX` = `4050110`
3. **prefix 直接当 effect prefix**，查 `cfg_system_effect[4050110+"01"]` = `405011001` = **"巅峰结构增强"**（EFFECT_ID=10，PARAM_LEVEL 有7行）
4. level=2 → 查 PARAM_LEVEL 第2行 = 200（万分比）

**关键特性**：
- 这些是**独立的巅峰专属强化项**（NAME 含"巅峰"，如"巅峰结构增强"/"巅峰武器单发"），与普通强化项（"龙骨结构增强"）不同，但同 EFFECT_ID，**叠加计算**。
- **不只结构值类**：涵盖结构(EID=10)/伤害(EID=12350)/暴击(EID=12030)/维修(EID=12050)等多类。
- **特定巅峰等级才解锁**（斗牛：巅峰5解锁slot02奖励，巅峰10解锁slot01奖励），N 随巅峰等级增长。
- optIdx=70/71 是巅峰奖励槽（**非"巅峰附带调校"**，§2.5 的旧描述已修正）。

**校验锚点**（斗牛40501）：

| 巅峰等级 | field[1] | 解析结果 |
|---|---|---|
| 5 | `405010270,2` | 巅峰结构增强 L2=200（万分比）|
| 10 | `405010170,1;405010171,1;405010270,3` | 强化充能功率 L1=2(伤害) + 强化脉冲聚焦 L1=2(伤害) + 巅峰结构增强 L3=300 |
| 20 | `405010170,3;405010171,3;405010270,5` | 伤害×2 L3=6 + 巅峰结构增强 L5=500 |

**实装代码**（`peakLevel.ts`）：
- `parsePeakRewardString(str)`：解析 field[1]
- `PeakRewardBonus`：按 EFFECT_ID 聚合（structurePermille/damagePermille/repairPermille）
- `computePeakBonus` 内部聚合 reward，`reward.structurePermille` 叠加到强化系数

## 二、调校系统（✅ 已定位 + 已实装）

### 2.1 机制（用户描述 + frida dump 实测确认）

调校是独立系统，三步解锁链：
1. **点永久加成属性槽**（无等级）→ 解锁
2. **纯消费槽**（无等级，消耗资源）→ 解锁
3. **调校槽**（上限10级，投武器技术提升）

前提：对应的普通强化项需要点过等级。
永久加成类型：结构值 / 伤害（看所在模块类型）。

### 2.2 数据载体（frida dump 实测，2026-07-03）

调校槽在 `cfg_system_enhance.json` 里，**optIdx=31-43**，完整字段：

| 字段 | 含义 | 示例 |
|---|---|---|
| `EFFECT_TYPE` | 效果类型（2=标准效果链） | 2 |
| `ADJUST_PROB` | **10级成功率**（递减） | (100,100,100,90,80,70,60,50,25,15) |
| `ADJUST_ENHANCE_INDEX` | 被调校的目标强化项 optIdx | 8 / 10 / 1 |
| `ADJUST_RARITY` | 所需武器技术稀有度 | 5 / 12 |
| `ADJUST_WEAPON_TECH_TYPE_LIMIT` | 武器技术类型限制 | "6,3" |
| `SYSTEM_EFFECT_PREFIX` | 调校效果前缀 | 192010 / 111211 |

**ADJUST_PROB 的10个值就是10级**（前3级100%，递减到15%），印证用户说的"上限10级"。

### 2.3 调校效果（从 PREFIX 查 system_effect）

| PREFIX | EFFECT_ID | 名称 | PARAM | 类型 |
|---|---|---|---|---|
| 111211 | 10 | 龙骨结构增强 | 700 | 结构值万分比 |
| 192010 | 13005 | 优势输出 | 5000 | 伤害类 |
| 191230 | 12014 | 精密射击 | 1015 | 命中/精度类 |

### 2.4 optIdx=20 是系统维修机制（非调校链）

optIdx=20（2299条）：`ENHANCE_COST=(0,)`，`ENHANCE_DEFAULT_LEVEL=1`，无 ADJUST 字段。
这是系统维修/损坏机制槽，不属于调校三步链。

### 2.5 optIdx=70/71 是巅峰附带调校（非主调校系统）

optIdx=70-71（606条）：EFFECT_TYPE=3/5，随巅峰等级到3级。
**★修正（2026-07-04）：这些是巅峰强化奖励槽（field[1]），不是"巅峰附带调校"。**
详见 §1.6。它们解析为独立的巅峰专属强化项（"巅峰结构增强"/"巅峰武器单发"等），
按 EFFECT_ID 叠加到面板。主调校系统是 optIdx=31-43（10级）。

### 2.6 武器技术资源

调校投入的"武器技术"来自 `cfg_weapon_tech.json`（62条），如：
- 601 基础脉冲炮技术（R2，火炮类）
- 806 轰炸机技术（R4，舰载机类）
- 807 平台轰炸技术（R4）

稀有度分级见 `cfg_weapon_tech_class.json`（8类：火炮/导弹/舰载机等）。

### 2.7 实装代码

- `packages/engine/src/data/tuneSystem.ts`：
  - `resolveTuneSystem(store, shipId)` 解析调校槽（含 targetEnhanceId 子系统 scope）
  - `computeTuneBonus(store, shipId, tuneLevels, enhanceLevels?)` 计算调校加成（含前提门控 + 多 EFFECT_ID 聚合）
  - `isTuneSlot(enhanceId)` 判断是否调校槽
- 校验脚本：`packages/engine/tests/data/tune-anchor.ts`（斗牛级3槽 + targetEnhanceId 锚点）
- 联动测试：`packages/engine/tests/data/tune-enhance-link.test.ts`（60501 10槽 + 门控 + 聚合）

### 2.8 ★调校与强化联动（2026-07-04 实装）

**前提门控**：调校生效需其目标强化项（targetEnhanceId）已点等级。
- targetEnhanceId = `shipId + slot + pad2(ADJUST_ENHANCE_INDEX)`（**同 slot scope**，子系统内解析）
- computeTuneBonus 传入 enhanceLevels，调校 level>0 但目标未点 → 计入 skipped[]
- 不传 enhanceLevels 时降级为不检查（向后兼容）

**效果接入面板**：调校应用自己的独立效果（不同 EFFECT_ID），按 EFFECT_ID 分发聚合：
- EID=10 → structureBonusPermille（结构万分比，叠加到强化系数作用于 skeleton）
- EID=13005 → damageBonusPermille（伤害）
- EID=12010/12014 → hitBonusPermille（命中）
- EID=12080/12081/12083 → interceptRate（拦截）
- EID=12050 → repairBonusPermille（维修）

resolveBlueprint 新增 `tuneLevels`/`enhanceLevels` 可选参数，自动聚合调校加成到面板。

## 三、强化项科技树与系统-模块容器语义（2026-07-04 实装）

### 3.1 ★系统-模块容器语义（核心）

**系统 = 容器，模块 = 内容。** 空容器（无模块）没有任何属性效果，等于系统不存在。

- 超主力舰有可选槽初始为空（60501 的 7 个空槽如舰首轨道炮/离子炮塔等），装配模块后才激活
- 数据层面：`cfg_system_enhance` 为**所有槽**（含空槽）预定义强化项，必须由运行时按"模块是否装配"门控
- `cfg_ship_slot` 只含已装配模块的槽，所以武器/装甲装配天然过滤了空槽；但 `resolveEnhanceSystem` 直接遍历 enhance 表，需显式门控

### 3.2 强化项依附系统槽

- enhanceId = shipId(5)+slot(2)+optIdx(2)，中间2位是系统槽
- 切换组每个成员是独立系统槽，各有独立强化树（6050110附加装甲 5项 vs 6050112主装甲 11项）
- 切换组切换时只激活当前选中成员的强化项

### 3.3 科技树前置依赖（cfg_system_enhance_tree.json）

格式 `{ enhanceId: ["parents;treeId", flag] }`：
- **parents**：前置 enhanceId 列表（逗号分隔），`0`=根节点（无前置）
- **treeId**：布局列号（同列纵向排列，UI 用）
- **flag**：节点类型（0=普通，1/2=特殊解锁节点）

特性：
- **严格 per-slot**：依赖从不跨槽，每个 (shipId, slot) 独立 DAG
- **只存解锁前置**（哪些 enhanceId 必须先获得），不存等级要求

示例（60501 slot01）：
```
optIdx01(根,treeId2) → optIdx02(treeId2)
optIdx03(根,treeId4) → optIdx04(treeId4) → optIdx06(treeId4) → optIdx09(treeId4)
optIdx10(根,flag1,treeId3) ┐
optIdx13(根,flag2,treeId3) ┼→ optIdx05(treeId3) → optIdx08(treeId3)
optIdx01(根,treeId2)      ┘
```

### 3.4 实装代码（enhanceSystem.ts）

- **加载**：`cfg_system_enhance_tree.json` → `store.systemEnhanceTree`（rawTypes/loader/nodeUtils/loadStore 4处）
- **EnhanceSlot 扩展**：`prerequisites`/`treeColumn`/`nodeFlag`
- **EnhanceSystemSlotInfo**（容器语义）：`hasModule`/`isSwitchable`/`isActive`/`installedModuleIds`
- **resolveEnhanceSystem(store, shipId, enabledSlots?)**：新增 enabledSlots 控制切换组激活
- **isEnhanceAvailable(slot, slotInfo, acquired)**：容器门控（有模块+已激活）+ 前置门控（前置全获得）双重判定
- **resolveEnhanceTree(store, shipId, slotId)**：按 treeColumn 分列，返回根节点
