## 二、对历史结论的勘误

WHITEBOX_RESULTS.md §1.4 写过一句："**字节码有自定义 opcode 重映射**（`loader.get_code()` 报 'Invalid Code'），但类的元数据完全可读"。

**这个推论是错的**，需要更正：

| 现象 | 旧解释（错误） | 实际真相（本次验证） |
|---|---|---|
| `loader.get_code(modname)` 报 "Invalid Code" | 内存中字节码 opcode 被重映射 | 仅**磁盘序列化格式**自定义；游戏 import 时还原成**标准 CPython 3.11 code object**，否则解释器跑不起来 |
| `co_names`/`co_consts` 可读 | "类的元数据完全可读" | 不止元数据，**完整字节码 + 反汇编 + marshal 全部可读** |

## 四、阶段0-B 已确认的客户端公式结论

### 4.1 已还原的源码：`get_ship_dps_calc`（line 1212 of `data/ship_attr_calc.py`）

```python
def get_ship_dps_calc(self, dps_type):
    calc = get_null_calculator()                                 # 1220

    def _add_new_calc(new_calc):                                 # 1222 (闭包)
        add_exp = new_calc.get_expression()
        add_exp = add_exp.to_int(add_exp)
        new_exp = calc.get_expression() + add_exp
        calc.make_merge(new_calc, new_exp)

    for module_info in self.cur_modules:                         # 1228
        module_id = module_info[0]
        slot_id   = module_info[1]
        if dps_type == common_definition.MA_SHIP_DPS:
            _add_new_calc(self.get_weapon_ship_dps_calc(slot_id, module_id))
        elif dps_type == common_definition.MA_SHIP_AIR_DPS:
            _add_new_calc(self.get_weapon_air_defend_dps_calc(slot_id, module_id))
        elif dps_type == common_definition.MA_SHIP_DESTROY_COEF_DPS:
            _add_new_calc(self.get_weapon_destroy_coef_dps_calc(slot_id, module_id))
        elif dps_type == common_definition.MA_REPAIR:
            _add_new_calc(self.get_module_repair_calc(slot_id, module_id))
        elif dps_type == common_definition.MA_SHIP_OPERATION:
            _add_new_calc(self.get_module_opeation_calc(slot_id, module_id))

    aircraft_group_num = Tb_cfg_ship.get(self.ship_id)[Tb_cfg_ship.AIRCRAFT_GROUP_NUM]   # 1242
    calc.cur_expression = calc.get_expression() * _C(aircraft_group_num)                 # 1244
    _add_new_calc(self.get_aircraft_dps_calc(dps_type, self.all_effects_list, self.cur_enhance_dic))  # 1248
    return calc
```

### 4.2 关键架构发现

**`C_*` / `V_*` 不是数值常量，是表达式树的字符串键名**：

```json
"C_BASE_NUM": "base_num"
"C_RATIO":    "const_ratio"
"V_ATTACK":   "attack"
"V_TOTAL_HURT":     "total_hurt"
"V_ATK_INTERVAL":   "attack_interval"
"V_ADDITIONAL_DPS_DPH": "additional_dps_dph"
"V_ACTION_TIMES":  "action_times"
"V_DURATION":      "duration"
"V_REPEAT_TIMES":  "repeat_times"
"V_ADD_DPS":       "additional_dps"
"V_ADD_NUM":       "add_num"
"V_ADD_RATIO":     "add_ratio"
"V_ADD_BASE_NUM":  "add_base"
```

这说明客户端 DPS 计算**不是直接算术**，而是构建一棵 `CalculationExpression` 表达式树：
- 每个武器/模块方法返回一个 `calc` 对象（含 `cur_expression`）
- `make_merge(new_calc, new_exp)` 把子表达式合并进父表达式
- `to_int()`、`get_expression()` 是表达式树的访问器
- 最终数值要 evaluate 这棵表达式树才得到

**含义**：要做精确的客户端 DPS 数值还原，必须同时 dump：
1. `CalculationExpression` 类（表达式树节点 / 求值语义）
2. 各子方法（`get_weapon_ship_dps_calc` 等）—— 它们构建叶子表达式
3. `_C()` 函数 —— 把整数包装成常量表达式节点

### 8.4 关键技术坑（备忘）

1. **frida 内联 PROBE_PY > ~7.5KB 时 `s.load()` 超时** —— V8 解析大字符串常量开销。解决：host 写 payload 文件，JS 跑 `exec(open(file).read())`，JS 体量恒定 1.5KB。
2. **Python 执行时间 > 60s（frida 默认 load 超时）** —— 不影响数据正确性，Python 跑完后所有文件正常落盘。下次可调大 frida 超时（`script.load()` 之后用更长的等待循环）。
3. **多次 frida attach 累积会让游戏崩溃** —— 单次跑完整 dump 比多次小探针更稳。
4. **`co_consts` / `co_names` 含 `<lambda>` / `<property>` 等带尖括号的名字** —— Windows 文件名禁用 `<>`，walker 必须 sanitize 路径。
5. **Python 3.13 的 `dis` 模块不能反汇编 3.11 code object**（API 变化）—— 必须在游戏解释器里跑 `dis.dis`，落盘文本。游戏外只做 marshal.loads 验证，不做反汇编。

### 8.5 模块别名机制（重要发现）

游戏启动时把一些深层模块注册成顶层名（`sys.modules` aliasing）：
- `configdata` → 实际是 `common.config.configdata`
- `common_definition` → 实际是 `common.common_definition`

公式代码里写 `configdata.AIRBORNE_WEAPON_HIT_RATE`，运行时 Python 通过 `sys.modules['configdata']` 找到。dump 时按真实路径存（`common.config.configdata/`）。
