# 客户端公式反编译专题

> 日期：2026-07-08
> 范围：从 infinite_lagrange_cn.exe 全量反编译 90,162 个 Python 3.11 code object 还原战斗公式
> 关联：`docs/14-客户端公式白盒.md`（阶段0 Frida 探针）、`data/client/DECOMPILED_DPS_FORMULAS.md`（公式结果）

---

## 一、目标

把客户端 `infinite_lagrange_cn.exe` 里**所有战斗相关公式**还原成可读 Python 源码，用于战斗模拟器。重点诉求：**火力值（DPS）计算公式**（`get_ship_dps_calc`、`get_weapon_attack_calc` 等）。

---

## 二、攻击面分析：客户端的三层反逆向保护

游戏的 Python 解释器是 CPython 3.11（自带 `python311.dll`）。在磁盘/内存里，所有 code object 都被**三层保护**改造过，标准工具直接打不开：

| 层 | 保护机制 | 表现 |
|---|---|---|
| L1 | 自定义 opcode 字节重映射 | 95 个 opcode 的字节值被换掉（如 std 135=MAKE_CELL，游戏里是 232） |
| L2 | 自定义 inline cache 数量表 | LOAD_GLOBAL 5 cache（std 4）、LOAD_ATTR 4 cache（std 9）、CALL 4 cache（std 3） |
| L3 | BINARY_OP 的 arg 子类型重映射 | `/` arg=26（std 11）、`/=` arg=27（std 24），其余不变 |

**额外障碍**：
- Python 3.12/3.13 的 `marshal.loads()` 在游戏 3.11 数据上**段错误**——标准库根本没法用
- `pycdc`（主流 3.11 反编译器）的 ASTree 缺 `POP_JUMP_FORWARD_IF_NONE/NOT_NONE`、`MAKE_CELL`、`COPY_FREE_VARS` 的 case
- `pycdc` 的 `bc_next` 不跳过 inline cache，依赖 cache 字节全为 0

## 五、阶段2：还原 3 张映射表

**核心思路**：游戏自己的 `dis.dis()` 会把 code object 正确反汇编成人类可读文本。我们拿 `marshal.bin`（自定义格式的字节）和 `disasm.txt`（标准助记符）做**交叉对照**，就能反推映射关系。

### 5.1 opcode 表（`game_opmap.json`）

脚本：`derive_opmap_v2.py`

```python
# 对每个 disasm 行 "  12  LOAD_GLOBAL  0"，取：
#   - offset = 12
#   - mnemonic = LOAD_GLOBAL
# 然后去 marshal.bin 取 code_bytes[12] 这个原始字节
# 聚合 {byte_value: {mnemonic: count}}，取众数
```

要点：
- 必须在 `Disassembly of <code object` 处截断，只保留**父级**指令——`dis.dis` 默认递归反汇编嵌套 code object，会污染统计
- 必须手写 marshal 解析（只取顶层 co_code）：CPython 3.12/3.13 的 `marshal.loads` 在游戏数据上段错误
- 结果：95 个 opcode，置信度全部 ≥ 0.95，绝大多数 = 1.0

### 5.2 cache 表（`game_cachemap.json`）

脚本：`derive_cachemap.py`

```python
# 对相邻两条指令 (off1, mn1) → (off2, mn2)：
#   delta = off2 - off1
#   cache_entries(mn1) = (delta - 2) / 2
# 按 mnemonic 聚合取众数
```

要点：
- 这步揭示了 L2 保护：LOAD_GLOBAL game=5 而 std=4，LOAD_ATTR game=4 而 std=9，CALL game=4 而 std=3
- cache 数差异是后续翻译器必须处理的核心难点之一

### 5.3 special arg 表（`game_special_ops.json`）

脚本：`derive_special_ops.py`

```python
# disasm 里 BINARY_OP 和 COMPARE_OP 行带符号注释：
#   BINARY_OP    26 (/)
#   COMPARE_OP   2 (==)
# 用正则提取 (game_arg, symbol)，对照 std 3.11 的 NB_* / cmp_type 表
```

要点：
- 大部分 arg 跟 std 一致，只有 `/`（26→11）和 `/=`（27→24）被换掉
- 这层保护最轻，但漏了会让所有除法表达式编译错误

## 六、阶段3：翻译器（核心难点）

输入：游戏自定义格式的 `.marshal.bin`
输出：标准 CPython 3.11 `.pyc`

### 6.1 v1：`translate_top_only.py`（顶层翻译）

只解析顶层 code object 的固定 header（TYPE_CODE + 5×4-byte int + TYPE_STRING + 4-byte len），翻译 co_code 后重打包。consts/names 等后续字段原样保留。

**结果**：90162 文件全部成功，但批量反编译时只有 96.6% 非空——剩下的 3.4% 是**含嵌套闭包**的函数（pycdc 会递归进入 co_consts 里的嵌套 code object，那些仍是游戏 opcode，撞上 "Unsupported opcode"）。

### 6.2 v2：`translate_recursive_v2.py`（递归翻译）★

正确做法：用 marshal walker 递归遍历**整个** marshal 树，遇到每个 TYPE_CODE 就翻译它的 co_code。

#### 关键坑 1：Python 3.11 code object 的字段顺序

CPython 3.11 的 marshal 写 code object 时，co_code 后面是 **10 个字段**，不是 9 个：

```
consts, names, localsplusnames, localspluskinds,    ← 4 个 object
filename, name, qualname,                            ← 3 个 object
firstlineno,                                         ← raw 4-byte long（不是 marshal object！）
linetable, exceptiontable                            ← 2 个 object
```

`firstlineno` 用 `w_long`（原始 4 字节）写，没有 TYPE_INT 前缀。把它当 object 解析会错位 1 字节，导致所有嵌套结构解析错乱。

#### 关键坑 2：jump 操作数必须重定位

inline cache 布局从 game 改成 std 后，**每条指令的字节长度都变了**。例如 LOAD_GLOBAL：game 5 cache = 12 字节，std 4 cache = 10 字节，每条少 2 字节。

相对 jump（`POP_JUMP_FORWARD_IF_*`、`JUMP_BACKWARD_*` 等）的 arg 是**相对偏移**，必须根据新布局重算：

```python
# Pass 1: 建立 game_offset → std_offset 的指令边界映射
g2s = {}
g, s = 0, 0
while g < n:
    g2s[g] = s
    game_op = code[g]
    gc = game_cache.get(game_op, 0)
    std_op = game2std.get(game_op)
    std_cache = STD_311_CACHE.get(std_op, 0) if std_op is not None else gc
    g += 2 + gc * 2
    s += 2 + std_cache * 2

# Pass 2: 翻译时，对每条 jump 重新计算 arg
if std_op in JUMP_FWD:
    target_game = g + 2 + arg * 2
    target_std = g2s.get(target_game)
    if target_std is not None:
        out_arg = (target_std - (s_now + 2)) // 2
```

**症状**：v1 翻译的 pyc 在 pycdc 反编译时报 `PycBuffer::getByte(): Unexpected end of stream`——因为 jump 越过了 co_code 末尾。

#### 关键坑 3：fallback 容错

walker 抛 `WalkAbort` 时自动回退到顶层翻译，避免一个坏文件卡死整个批量。

#### 结果
- **90134/90162 递归成功（99.97%）**
- 26 fallback、2 fail，全是调试渲染代码（TYPE_LONG 大整数常量，与公式无关）
- 总共翻译了 **97262 个 code object**（含嵌套）

## 七、阶段4：patch pycdc + 批量反编译

### 7.1 为什么必须 patch

pycdc（zrax/pycdc）的 opcode **表**认识 Python 3.11 的 MAKE_CELL/COPY_FREE_VARS/POP_JUMP_FORWARD_IF_NONE/NOT_NONE，但 ASTree **反编译器 switch** 没有对应的 case，落到 `default:` 就报 `Unsupported opcode` 直接退出。

例：`get_ship_dps_calc` 第一条就是 `MAKE_CELL`（建立闭包 cell），v1 翻译版 pycdc 报：
```
Unsupported opcode: MAKE_CELL (225)
```
（225 是 pycdc 内部 enum 值，对应 std byte 135）

### 7.2 patch 内容（`patch_pycdc.py`）

对 `~/pycdc/ASTree.cpp` 做精确文本替换：

1. **POP_JUMP_FORWARD_IF_NONE / NOT_NONE**：加到与 IF_FALSE/IF_TRUE 共享的条件跳转 case 列表（含 "Pop condition before the jump"、"offset is relative" 两组判断）
2. **MAKE_CELL / COPY_FREE_VARS**：加到 PRECALL/RESUME 旁边的 no-op case（这俩是闭包实现细节，对反编译源码无影响）

### 7.3 pycdc 工作机制要点

- `bc_next()` 一次读 2 字节（opcode + arg），**不主动跳 cache**
- cache 字节全 0 时会被当 CACHE opcode 读取，ASTree 里 `case Pyc::CACHE: break;` 当 no-op 处理
- 所以翻译器必须把所有 cache 填 0（`translate_recursive_v2.py` 已这么做）
- MAGIC 必须是 `\xa7\x0d\x0d\x0a`（Python 3.11，3495）

### 7.4 批量反编译（`batch_decompile_v2.py`）

`ProcessPoolExecutor` + 12 worker + `chunksize=200`，每个文件用 `subprocess.run([pycdc, pyc], timeout=10)`。

**结果**：
| 指标 | v1（顶层翻译） | **v2（递归翻译 + patch）** |
|---|---|---|
| 成功反编译 | 87,082 (96.6%) | **87,987 (97.6%)** |
| 空输出 | 3,080 | **2,173** ↓29% |
| 输出总量 | 30 MB | **41.8 MB** ↑39% |

最大的改进不是数字，而是**关键公式从空变成完整源码**。

## 十一、关键经验

1. **解释器自带的 dis 是最强线索**：游戏再怎么改字节布局，自己的 `dis.dis()` 必须能正确反汇编。拿 disasm 文本反推映射表，比逆向解释器二进制省 100 倍力气。

2. **三层保护的设计意图**：L1 改字节值（让 pycdas 直接看不懂）、L2 改 cache 布局（让重打包后指令错位）、L3 改 BINARY_OP arg（让除法表达式错）。每一层单独都能让标准工具失效，组合起来相当有迷惑性。

3. **marshal 字段细节是最大坑**：CPython 不同版本 marshal 格式微调（如 3.11 加 exceptiontable、firstlineno 用 raw long 不用 object）。靠经验猜字段数会错，必须查 `Python/marshal.c` 源码。

4. **反编译器缺 case 比 opcode 表缺项更难发现**：pycdc 报 "Unsupported opcode: MAKE_CELL" 但其实 MAKE_CELL 在表里。错误信息误导性强，需要去 ASTree.cpp switch 里数 case 才能定位。

5. **jump 重定位是必须的**：cache 布局变化导致相对偏移失效，这个 bug 会让所有有跳转的函数反编译出空结果，但不会报错——非常隐蔽。

6. **fallback 容错保批量稳定**：90162 文件里有几十个边缘 case（TYPE_LONG、超深嵌套），与其为它们写完整支持，不如直接 fallback 到顶层翻译，损失 0.03% 换取批量稳定性。
