## 二、逐函数对齐结果（6 函数）

### 1. calc_effect_add ↔ getEnhanceAdd（effectList.ts:141）— ✅ 已对齐
- 三通道 switch（ratio_add/num_add/base_num_add ±）与 Python 一致。
- **EFFECT_SHIP_HP(=10) /100**：Python 末尾 `if effect_id==SHIP_HP: ratio=100` 把三通道除 100。TS 用 **‱/10000** 约定（`structureBonusPermille += value`，`/10000` 应用），与 Python 的 `%→/100` **数值等价**（`base*(100+param/100)/100 == base*(1+param/10000)`）。结构测试 cv3000-structure + module-structure 共 6/6 通过。
- 等级缩放 `PARAM × cur_level × 1.0/max_level`（浮点除）：TS 在 resolveBlueprint 阶段预缩放进 EffectEntry.value，一致。

### 2. get_module_effect_real_value（单位变换表）— ✅ 已对齐（无重复变换）
Python 是唯一的单位变换总表（%100/%1000//100//5000/取负）。TS 的两条路径：
- **模块原值路径**（modDamageInc 等）：`getAllModuleEffects` 取原始 PARAM，`moduleStructureBonus = base×param/10000` = Python 的 `base×(param/100)/100`，等价。
- **强化路径**（effectList）：值经 calc_effect_add，SHIP_HP 已含 /100（见上）。
- 确认无重复变换。EFFECT_DAMAGE_INC(12020)/AIRCRAFT_INC(12062)/DESTROY_INC(12060) 的 PARAM 在 weaponNumAttr 是 ratio_add，TS 装配时取原值绑 V_SKILL_EFFECT_RATIO，evalAttack 不再 /100，正确。

### 3. collect_skill_effect_ratio ↔ 防空 skill_ratio 通道 — ✅ 已修复（commit 22c4d6b）
**本次唯一实质性修复**。Python 对 MA_MODULE_AIR_DPS 在 duration/cd 上把模块原值加进 skill_effect_ratio：
```
duration: skill_ratio += DURATION_INC(12310) + DURATION_DEC(12311→取负)
cd:       skill_ratio += CD_TIME_INC(12305)   + CD_TIME_DEC(12306→取负)
→ 进 (100+add_ratio+skill_ratio)/100
```
旧 TS 只处理 DEC(12306/12311) 作 `airCdReduction/airDurReduction` 从 addRatio **减**——**漏了 INC**（12305/12310）且符号约定不同。修复：
- 字段改名 `airCd/DurReduction → airCd/DurSkillRatio`，值 = `INC_param − DEC_param`（带符号）
- `evalEnhanceBasic` 补 `skillRatio` 参数；duration/cd 通道改为**加** skill_ratio
- 清除 effectList.ts:54 TODO
- 数值等价验证：仅 DEC 时 `−DEC == +(−DEC)`，firepower-anchor 16/16 全过（13124=1440、13122=375、13011防空=1851）

### 4. get_after_system_effect_value ↔ evalEnhanceBasic — ✅ 已对齐
Python 主公式 `ret = (base+Σadd_base)×(100+Σadd_ratio+skill_effect_ratio)/100.0 + Σadd_num`。
- evalAttack（attack 通道）早已含 skillRatio。
- duration/cd 通道经 #3 修复后含 skill_ratio（flight/repeat 无 skill_ratio，与 collect_skill_effect_ratio 的 attr_type 分支一致）。
- `/100.0` 浮点除：TS 用 JS 默认浮点，一致。

### 5. get_module_attr_value ↔ attr_type 分发表（getAllModuleEffects + resolveBlueprintPanel）— ✅ 已对齐
- TS 的 getAllModuleEffects 已处理 10033/10021/10/12020/12010+12012/1/2/12030/12050 等，覆盖 Python MA_ARMOR→EFFECT_BALLISTIC_INJURY_SUB、MA_SHIELD→EFFECT_ENERGY_INJURY_DEC 等核心 EID。
- ms↔s 转换（ATTACK_INTERVAL/DURATION/CD_TIME/FLIGHT_TIME 都 /1000，部分 max(.,0)）：TS 的 computeWeaponDps 已 `*1000/1000` 处理，一致。

### 6. is_valid_effect ↔ isValidEffect（effectList.ts:103）— ✅ 核心逻辑 100% 对齐
反编译为空但有完整 disasm（375行）。对照 `is_valid_effect.disasm.txt`：
- TARGET_MODULE_TYPE 解码常量 `10000/100/10` 与 TS `Math.floor(tmt/10000)` / `Math.floor(tmt/100)%10` / `tmt%10` 完全一致（disasm co_consts_simple 确认）
- WEAPON_FILTER_ENERGY 分支（filterType==10000 → energy）与 disasm 行 306-340 一致
- effect_module_type/effect_weapon_type 匹配与 disasm 行 346-367 一致
- TARGET_SYSTEM→TARGET_INDEX→TARGET_MODULE_TYPE 三段式与 disasm 顺序一致
- **未实现**：TARGET_SHIP 的 drone/fighter/boat 分支（disasm 行 3590-3650）。这些仅在判定**无人机/战机/艇自身**的作用域时触发，面板火力计算（舰船算自身武器，target_ship 自指）不涉及，非阻塞。
