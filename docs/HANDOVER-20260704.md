## 一、本次完成的工作

### 1. prebundle 崩溃修复（dev server 启动）
**现象**：`npm run dev:h5` 崩溃在 `webpack-virtual-modules` 的 `_writeVirtualFile is not a function`
**根因**：`config/index.ts` 把 `prebundle: { enable: false }` 写在**顶层**，但 `webpack5-runner` 的 `getPrebundleOptions()` 只读 `config.compiler.prebundle`（`Combination.js:103`），且仅当 `compiler` 是对象时才合并。
**修复**：`packages/editor/config/index.ts` 改为 `compiler: { type: "webpack5", prebundle: { enable: false } }`

### 2. node_modules 软链重建
**根因**：`node_modules/@lagrange/engine` 是**旧实体目录拷贝**（非 symlink），导致所有引擎改动（切换组修复、weapon_priority 加载）都失效——页面读的是 7月2日的旧 dist。
**修复**：删除旧拷贝，`npm install` 重建 workspace symlink → `node_modules/@lagrange/engine → packages/engine`

### 3. loadWeaponPriority 导出修复
**根因**：`export *` 在 TS/webpack 静态分析下丢失 `loadWeaponPriority` 具名值导出 → editor 无法加载 weapon_priority 表 → **火力计算全部归零**。
**修复**：`packages/engine/src/index.ts` 显式补导出 `export { loadWeaponPriority } from './data/blueprintCalc.js'`

### 4. 模块切换功能（超主力舰核心功能）★
**语义重构**：之前用 `ADDITIONAL_SYS` 字段判断"固定/可选"，但按用户澄清——**固定只是初始默认项，同组所有模块（含固定项）等价可选**。真正判定标准是**同 GROUP 成员数**：
- 同 GROUP ≥2 模块 = 切换组（全部等价可选）
- 单模块组 = 固定不可换

**改动**：
- `blueprintSelector.ts` `getShipSystems()`：新增 `isSwitchable`/`isDefault` 字段，`enabled` 完全由 `enabledSlots` 驱动
- `blueprintCalc.ts` 新增 `resolveEnabledSystems()` 统一判定（引擎层切换组互斥，避免固定项与可选项双重装配）
- `index.tsx` `toggleModule`：改用 `isSwitchable`，固定项也能切换
- 模块号改为 **动态分配**：按可切换组 GROUP 数值升序，依次 M/A/B/C/D...（不再静态映射）

## 二、关键技术发现

### 引擎层与 UI 层的语义对齐 ★
模块切换功能涉及**引擎层（blueprintCalc）和 UI 层（blueprintSelector）两套独立逻辑**，必须用统一的"启用系统集合"判定，否则会出现：
- UI 显示切换了，但引擎仍装配旧模块（双重计算）
- 火力/防御数值不联动

`resolveEnabledSystems()` 是两层的桥梁，规则与 selector 完全一致。

### node_modules workspace 软链陷阱
npm workspaces 应该创建 symlink，但某些情况下会生成实体拷贝。**改了 packages/engine 但 dev server 无响应时，第一时间检查 `node_modules/@lagrange/engine` 是否 symlink**：
```bash
node -e "console.log(require('fs').lstatSync('node_modules/@lagrange/engine').isSymbolicLink())"
```

### 端到端验证原则
**不能只验证中间层（武器数），必须验证页面实际显示的数值（火力/防御）**。之前连续两轮"验证通过"都是假象——因为只验了 `resolveShipWeapons` 的武器数，没验 `resolveBlueprintPanel` 的火力值（一直是 0）。

### neox 引擎私有纹理格式
`get_ship_blueprint_model_path` 返回的 .png 文件实际是 neox 私有纹理格式（magic `c1 59 41 0d`），Pillow 无法读取。正面卡片图目前只能靠 B站Wiki。

**根因**：上一轮把 `cfg_module_effect` 的 EFFECT_ID=10 误判为"战后修复"，实际它是**舰船结构值提高（万分比）**，与抵抗(EID=10033)/护盾(EID=10021)完全平行的模块基础属性。

**分层模型**（用户确认）：
```
Layer0 baseStructure（cfg_ship[4]，出厂骨架，代码不改）
Layer1 moduleStructureBonus（模块 EID=10 万分比 × base）→ skeleton
Layer2 强化(permille)×skeleton + 巅峰(绝对值) + 技术值(绝对值) → final
```

**改动**：
- `blueprintCalc.ts`：`getBaseDefense` 加 `structurePermille`；`BlueprintPanel` 加 `baseStructure`/`moduleStructureBonus`
- `blueprintResolver.ts`：`resolveBlueprint` 加 `moduleStructureBonus?` 参数，强化作用于 skeleton

## 七、追加：巅峰强化奖励 + 调校联动 + 强化科技树（2026-07-04 续2）

### 1. 架构澄清：蓝图数据层 vs 战斗实例层 ★
- **蓝图数据层**（source of truth）：蓝图本身定义（模块/强化/调校/巅峰），战斗不影响它
- **战斗实例层**（derived）：客户端从蓝图聚合战斗数值，传服务器，结果不回写蓝图
- `resolveBlueprint` = 蓝图态→聚合数值的转换器（模拟客户端聚合）
- 详见 `docs/13` §〇

### 2. 巅峰强化奖励 field[1] ★
`cfg_ship_peak_level[shipId+peakLv]` 是2元素数组：
- field[0] = 普通强化项等级给定（EID=12 结构绝对值，已有）
- **field[1] = `"enhanceId,level;..."` 巅峰专属强化项等级给定**（新增，optIdx=70/71）

field[1] 解析链：enhanceId → `systemEnhance[enhanceId].SYSTEM_EFFECT_PREFIX`（当 effect prefix）→ `systemEffect[prefix+"01"]` = "巅峰结构增强"/"巅峰武器单发"等独立强化项 → PARAM_LEVEL 第 level 行。涵盖结构(EID=10)/伤害(EID=12350)/暴击(EID=12030)/维修(EID=12050)多类。

实装：`peakLevel.ts` 的 `parsePeakRewardString` + `PeakRewardBonus`，结构加成叠加到 structureBonusPermille。bull-panel 测试期望值从 45948 修正为 46669（含 field[1] 的 2%）。

### 3. 调校强化联动 ★
- **前提门控**：调校槽(optIdx31-43)的 `targetEnhanceId = shipId+slot+pad2(ADJUST_ENHANCE_INDEX)`（同 slot scope）。调校生效需目标强化项已点等级
- **效果接入面板**：调校独立效果按 EFFECT_ID 分发（结构/伤害/命中/拦截/维修），接入 resolveBlueprint

实装：`tuneSystem.ts` 的 resolveTuneSystem 加 targetEnhanceId，computeTuneBonus 加门控+多EFFECT_ID聚合。

### 4. 强化项科技树 + 系统-模块容器语义 ★

**容器语义**（关键）：系统=容器，模块=内容。**空槽（无模块）的强化项不生效**（系统不存在）。超主力舰有可选槽初始为空（60501 的 7 个空槽），装配模块后才激活。`cfg_system_enhance` 为所有槽（含空槽）预定义强化项，必须运行时门控。

**科技树前置依赖**（`cfg_system_enhance_tree.json`，新加载）：
- 格式 `{ enhanceId: ["parents;treeId", flag] }`，parents=前置列表（`0`=根），treeId=布局列号，flag=节点类型
- 严格 per-slot（无跨槽依赖），只存解锁前置（不存等级要求）

**强化项依附系统槽**：切换组每个成员是独立系统槽，各有独立强化树。

实装：
- 加载 `cfg_system_enhance_tree.json`（rawTypes/loader/nodeUtils/loadStore 4处）
- `enhanceSystem.ts`：EnhanceSlot 加 prerequisites/treeColumn/nodeFlag；新增 `EnhanceSystemSlotInfo`（hasModule/isActive 容器语义）；新增 `isEnhanceAvailable`（容器+前置双重门控）、`resolveEnhanceTree`（按列分组）
- `resolveEnhanceSystem` 新增 enabledSlots 参数（切换组激活控制）

## 八、追加：装配科技点 + 超主力舰版本号结构加成（2026-07-04 续3）

### 1. 装配科技点机制 ★
**总科技点 = 强化项消耗点 + 装配点**，两者独立计入：
- **装配点**（仅超主力舰）：初始为空的系统槽首次装配模块 +10点/切换组（同组切换不另收费）
- **强化项消耗点**：所有强化项 ENHANCE_COST 累计（**不区分效果是否生效**——未激活系统的强化项投入也计入）

★关键澄清：**效果生效 vs 科技点计入是两件独立的事**。
- 效果生效：看模块装配/激活（容器语义门控）
- 科技点计入：强化项被点就计入（不门控）

★**科技点 vs 结构是两个独立维度**。科技点（装配点+强化项消耗）本身要记录保留，不管是否影响结构。
- 科技点的**一个用途**是版本号结构加成（仅普通舰生效）
- 超主力舰科技点照常统计，但其结构加成=0（不影响结构值）
- 科技点统计能力（countInstallTechPoints / countTechPoints）独立于结构计算，保留供 UI 显示等用途

★"初始为空"是运行时状态：配置表无法可靠判定哪些系统初始为空（CV3000 反例：护航艇系统有 POINT_REQUIRED 但初始可能有模块）。因此装配点由**调用方传入** `installPoints`（玩家装配记录），引擎不自动推断。`countInstallTechPoints` 仅供启发式推测（ADDITIONAL_SYS=1 && MAIN≠1）。

### 2. 版本号结构加成分舰级 ★
- 普通舰（护卫/驱逐/巡洋）：`cfg_ship_type[9]`（驱逐=40/护卫=20/巡洋=60）
- ★**超主力舰（战巡/航母/战列/支援）无版本号结构加成**——cfg_ship_type[9]=0
  - 实测印证：CV3000(航母) finalStructure=278340×1.33=370192，无版本号加成
  - cfg_ship_type[10]=50 不是结构加成用途（待定）
- 公式：版本号加成 = (techPoints + installPoints) × shipHpAdd（超主力舰 shipHpAdd=0 → 加成=0，但科技点仍统计）

### 3. 实装
- `blueprintResolver.ts`：
  - `lookupShipHpAdd`：超主力舰(6/7/8/9)用 cfg_ship_type[10]，普通舰用[9]
  - `countInstallTechPoints(store, shipId)`：启发式推测（导出，非精确）
  - `BlueprintOptions.installPoints`：调用方传入确切装配点（不传默认0）
  - 版本号公式：`(techPoints + installPoints) × shipHpAdd`
- `rawTypes.ts`：ship_type [9]=普通舰ship_hp_add，[10]=超主力舰ship_hp_add_super(50)

### 4. 测试

### 1. 受维修量提升属性 ★
公式：`受维修量提升% = 装甲抵抗值 × 舰种系数`
- 装甲抵抗值 = 模块EID=10033 + 强化EID=10033（不含ship_type基础抵抗）
- 系数按舰种（cfg_ship[11]）：护卫0.05/驱逐0.1/巡洋0.2/超主力0.25/战机护航艇0
- 实测验证：斗牛(驱逐0.1) 模块20×0.1=2% + 强化16×0.1=1.6% = 3.6% ✓
- 天枢(航母0.25) 抵抗390×0.25=97.5% ✓
