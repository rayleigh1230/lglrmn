### C. 火力公式结构对齐反编译源码

`computeFirepower` 从两通道改成**三通道**结构，与 `AttackCalculator.expression` 完全对齐：

```
反编译：attack = (C_RATIO/100) × ((base + V_ADD_BASE_NUM) × (100 + V_ADD_RATIO)/100 + V_ADD_NUM)

引擎 ：effectiveDph = (dph + baseDamageBonus) × (1 + weaponDmgBonus) + flatDamageBonus
                       └── V_ADD_BASE_NUM ──┘     └─ V_ADD_RATIO ─┘   └─ V_ADD_NUM ─┘
                            (会被放大)                (万分比)          (不被放大)
```

**行为不变性**：`flatDamageBonus` 初始为 0，`(x)(1+y) + 0 ≡ (x)(1+y)`，对现有 4 武器 10 锚点数据**零数值影响**。未来发现属于 V_ADD_NUM 的 EID 时，直接填此字段即可。

## 四、客户端存档模型（背景知识，给后续工作参考）

反编译发现的**服务器存档结构**：

**ShipField 战斗相关**（57 项中战斗用到的）：
- 核心标识：SHIP_ID_U / SHIP_ID / USERID / TEAM_ID
- 装配数据：MODULES（武器模块串）/ EFFECTS_ENHANCED / AIRCRAFTS / DRONES
- 蓝图：BP_ID / SUPER_MAIN_SHIP_BP_SYSTEM_SCHEME_* / EFFECTS_PEAK_LEVEL
- 状态：HP / HP_MAX / STATE / BATTLE_STATE

**TeamField 战斗相关**（150+ 项中战斗用到的）：
- 核心标识：TEAM_ID / USERID / NAME / TEAM_TYPE / FLAGSHIP_ID_U
- 舰船成员：SHIP_INFO_NEW（编队舰船清单）
- 行动指令：ACTION / PLAN_ID / TASK_TYPE / ATTACK_TARGET
- 属性位串：ATTR_STR（一个字符串编码多个布尔位）

**关键序列化格式**：
- `MODULES` = `parse_cfg_str_to_list_of_list` → `[[slot,wid,...]]`
- `EFFECTS_ENHANCED` = `"slotId,subType,techId,level,..."` 三元组
- `AIRCRAFTS` = `parse_cfg_str_to_dict_of_list` → `{key:[v1,v2]}`
- `SHIP_INFO_NEW` = Python list 字面量

**关键洞察**：ship_id 不是数据包而是数据库主键。开战协议只发 level_id/team_id，服务器从自己库里查全部数据后自行计算战斗。所以模拟器无法"造舰队塞给服务器算"，必须**自行实现战斗计算**——这也是反编译工程的核心意义。

## 八、关键经验

1. **反编译核心思路**：游戏自己的 `dis.dis()` 是最强线索——游戏再怎么改字节布局，自己的反汇编器必须能正确反汇编。拿 disasm 文本反推映射表，比逆向解释器二进制省 100 倍力气。

2. **数据模型对齐**：客户端用字符串（MODULES/EFFECTS_ENHANCED）是因为网络序列化省带宽，模拟器没必要跟——存结构化对象，codec 层负责边界转换。`enhanceLevels: Record<string, number>` 就是这个抽象的正确范例。

3. **公式结构 vs 经验校准**：4 武器 10 锚点 0 误差 ≠ 公式结构对齐。经验校准在"恰好不触发缺口"的数据上会骗人。结构性对齐（三条通道）比数值匹配更可靠。

4. **行为不变的公式改造**：加 `+ 0` 形式的新通道，数值上完全等价于旧公式，但结构上预留了未来扩展点。这是低风险改造的范式。
