# 蓝图与舰队配置系统 —— 设计文档

> 日期：2026-07-02
> 状态：待批准
> 目标：搭建「舰船蓝图系统（可加技术点/自由装配）+ 舰队配置系统（自由组队）」，作为战斗模拟器的前置数据层，让后续模拟测试能快速配置场景而非手工硬编码数值。
> 前置文档：`docs/10-客户端配置表字段字典.md`、`docs/09-固定种子机制破解.md`

---

## 一、背景与动机

### 1.1 当前痛点

战斗模拟器引擎核心已校准（命中率公式、伤害模型、种子机制），但**机制只跑通几个极简场景**。根因是测试侧的数据配置方式：

```typescript
// 现状：每个测试文件顶部硬编码魔法数字
const DPH = 1200;           // 康纳马拉主炮单发
const RESISTANCE = 270;     // CV3000抵抗
const HP = 370192;          // CV3000含buff的HP
```

换个配置（不同蓝图、不同技术点、不同船型组合）就要重新查表、重新硬编码，导致：
- 无法快速验证新场景
- 配置与引擎耦合，改一处数值要改多个测试
- 无法做"自由组队 vs 自由组队"的批量对比

### 1.2 数据链路断层（核心瓶颈）

战报里的蓝图科技串（`techId=8206,1132,1210`）需要映射到 EFFECT 才能算出数值，但这个映射表未 dump：

```
战报科技串 techId(8206) → [Tb_cfg_weapon_tech 缺失!] → EFFECT(伤害+X%)
```

没有这个映射，蓝图系统装出的船**算不出真实 dph/命中率**，又会退回硬编码。

### 1.3 目标

搭建一个本地 Web UI 编辑器，支持：
1. 选船型 → 装蓝图（多模块+等级，完整复刻游戏机制）
2. 自由组队（双方各摆 N 艘，含阵型）
3. 一键调用引擎模拟 → 展示结果（HP曲线/伤害统计）

## 二、整体路线图（三阶段，严格顺序）

| 阶段 | 内容 | 产出 | 执行方 |
|---|---|---|---|
| **① 补数据** | dump 缺失的 tech/module 表 + 验证 techId→EFFECT 链路 | 完整配置数据集 | 用户操作游戏 + AI改脚本 |
| **② 数据层+引擎对接** | `clientData` 加载器 + `resolveBlueprint(科技串)→Ship` | 引擎能用真实数据跑 | AI |
| **③ Web UI 编辑器** | 选船→装蓝图→组队→模拟 | 可视化配置入口 | AI |

**阶段①是硬前置**：没有 techId→EFFECT 映射，后续都是空中楼阁。

## 三、阶段①：补 dump 缺失数据

### 3.1 需要补 dump 的表

经扫描 `modules/03_ship_attribute.txt`，确认以下表与战报 techId 链路相关：

| 表名 | 推断用途 | 优先级 |
|---|---|---|
| `Tb_cfg_weapon_tech` | **武器科技节点定义**（techId → 效果组） | ★★★ 核心 |
| `Tb_cfg_weapon_tech_pack` | 科技包（多个 techId 打包） | ★★★ 核心 |
| `Tb_cfg_module` | 模块定义（模块类型/槽位规则） | ★★ |
| `Tb_cfg_weapon_tech_class` | 科技分类 | ★ |
| `Tb_cfg_module_blueprint` | 模块蓝图（解锁关系） | ★ |

### 3.2 实施方式

在现有 `data/client/tools/dump_all_blueprint.py` 的 `dump_table` 调用列表末尾追加：

```python
# 11. 武器科技(techId→效果映射，核心缺口)
dump_table('common.config.db.Tb_cfg_weapon_tech', 'cfg_weapon_tech.json')
# 12. 科技包
dump_table('common.config.db.Tb_cfg_weapon_tech_pack', 'cfg_weapon_tech_pack.json')
# 13. 模块定义
dump_table('common.config.db.Tb_cfg_module', 'cfg_module.json')
# 14. 科技分类
dump_table('common.config.db.Tb_cfg_weapon_tech_class', 'cfg_weapon_tech_class.json')
# 15. 模块蓝图
dump_table('common.config.db.Tb_cfg_module_blueprint', 'cfg_module_blueprint.json')
```

### 3.3 用户操作步骤

1. 启动游戏客户端（`infinite_lagrange_cn.exe`）
2. 管理员运行：`python data/client/tools/dump_all_blueprint.py`
3. 脚本输出到 `E:\星际猎人\dumped\`（脚本内 OUTDIR，按 FRIDA_DUMP_GUIDE.md 适配本机路径）
4. 将新表拷贝到 `data/client/config/`

### 3.4 验证标准（AI 负责）

dump 完成后，AI 验证 techId→EFFECT 链路是否打通：

```
战报科技串: 8020104,20,8206,1  (CV3000 主炮槽)
  → techId=8206 查 cfg_weapon_tech
  → 得到 EFFECT_ID + PARAM
  → 与 cfg_module_effect / cfg_system_effect 对应
  → 能算出该武器 dph/命中率
```

**验证锚点**：CV3000 基础结构 278340，战报实测 370192 = 278340 × 1.33。若新表能解释这个 +33% 的来源（结构强化 EFFECT），则链路确认打通。

## 四、阶段②：数据层 + 引擎对接

### 4.1 数据层架构（`packages/engine/src/data/`）

```
packages/engine/src/data/
├─ clientData.ts        # 配置表加载器（读 data/client/config/*.json）
├─ types.ts             # 客户端数据类型（原始字段）
├─ shipResolver.ts      # shipId → Ship（船型骨架）
├─ weaponResolver.ts    # weaponId → WeaponSystem（武器循环）
├─ blueprintResolver.ts # ★ 科技串 → 完整 Ship（含数值）
└─ index.ts             # 公共 API
```

### 4.2 核心函数

```typescript
// 1. 加载所有配置表（启动一次）
loadClientData(configDir): ClientDataStore

// 2. 船型骨架（无数值，来自 cfg_ship）
resolveShipHull(shipId): ShipHull

// 3. 武器循环（来自 cfg_weapon）
resolveWeapon(weaponId): WeaponSystemDef

// 4. ★ 蓝图解析（核心：科技串 → 引擎 Ship）
resolveBlueprint(techString: string): Ship
//   解析 "slotId,subType,techId,lv,..." 
//   → 查 cfg_weapon_tech 得 EFFECT
//   → 汇总成 dph/hitRate/resistance/shield
//   → 输出引擎可用的 Ship 对象

// 5. 战报配置导入（直接用真实战报）
importFromBattleReport(reportDir): { ally: Fleet, enemy: Fleet }
```

### 4.3 EFFECT_ID → 引擎属性转译规则

基于 `docs/10-客户端配置表字段字典.md` 第四节：

| EFFECT_ID | 引擎字段 | 转译 |
|---|---|---|
| 12156/12010 | `hitBonus` | PARAM/100 |
| 12030/12031 | `critRate` | PARAM/100 |
| 10010 | `dodge` | PARAM/100 |
| 10（结构强化） | `structure` | base × (1+PARAM) 或 base+PARAM |
| 13025（能量护盾） | `shield` | PARAM/100 |
| 12300/12020 | `dph` | +PARAM 或 ×(1+PARAM) |

具体加法/乘法规则待阶段①拿到真实 tech 表后确认。

### 4.4 引擎对接验证

用现有战报（`battle_report_today1`，CV3000 vs 2混沌）回测：
- `importFromBattleReport` 导入配置 → `resolveBlueprint` 算出双方 Ship
- 引擎模拟 → 输出逐5秒 HP 曲线
- 与战报块1的 HP 曲线对比（统计等价，非比特复现，见 09 文档新范式）

## 五、阶段③：Web UI 编辑器

### 5.1 技术选型

- **框架**：React + Vite（轻量，本地运行）
- **引擎调用**：引擎编译成 JS → Web Worker 跑模拟（避免阻塞 UI）
- **位置**：`packages/editor/`（monorepo 新 workspace）

### 5.2 核心交互流程

```
[选船型] → [装蓝图：槽位×模块×等级] → [预览数值] 
                                            ↓
[我方舰队编排：拖拽组队+阵型] ← [敌方舰队编排]
                                            ↓
                              [一键模拟] → [结果展示]
```

### 5.3 页面结构

1. **蓝图编辑器**：左侧船型树（787艘）+ 右侧槽位面板（每个槽位可装多模块，下拉选 techId + 等级滑块）+ 底部实时数值预览（dph/HP/命中/抵抗）
2. **舰队编排器**：双方各一个舰队面板，从蓝图库拖拽舰船入队，设置数量和阵型格位
3. **模拟结果页**：HP曲线图（双方逐5秒）+ 伤害统计 + 命中分布

### 5.4 配置持久化

舰队配置可存为 JSON 文件（`data/fleets/*.json`），可复现、可分享、可版本管理。

## 六、范围控制（YAGNI）

**本阶段不做**：
- 舰载机系统（引擎接口已留，但 UI 不暴露）
- 反导/拦截的 UI 配置（机制未校准）
- 子系统摧毁的可视化
- 多人协作/云存储（纯本地）
- 真实游戏美术资源（用文字/简图）

**本阶段必须做**：
- 选船 + 装蓝图（多模块+等级） + 组队 + 模拟 + 结果展示
- 配置存盘/读盘
- 战报配置导入（用真实数据校准）

## 七、风险与缓解

| 风险 | 缓解 |
|---|---|
| dump 出的表字段含义不明 | 阶段①专门做字段解析验证，用 CV3000 的 +33% HP 锚点反推 |
| techId→EFFECT 是多对多（一个 tech 含多个 EFFECT） | blueprintResolver 做汇总而非单查，按 EFFECT_ID 归类叠加 |
| EFFECT 的加法/乘法规则不确定 | 用多个已知战报交叉验证（today1/today2/nobuff 三组对照） |
| UI 开发量大拖延 | 阶段③可拆：先做蓝图编辑器（能用），再做舰队编排，最后做结果可视化 |
