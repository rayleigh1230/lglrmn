## 〇、设计视角的根本转变（v2核心修正）

**v1（已废）出发点**：设计者刻意设计"确定性/固定种子"。
**v2（本版）出发点**：设计者首要考虑服务器压力，**确定性是省算力的副产品**。

策划的关键洞察：
1. 设计者首要目标是**减少运算量**（服务器压力），不是"确定性"
2. "N≥2固定种子现象"是极端场景（10同型打1靶），很少发生
3. 大部分场景是**多舰种大编队混战**，设计者优化的是这个
4. 固定种子是**服务器压力下的妥协**，不是炫技

这个转变彻底改变了方案选择（见§九的方案演进）。

## 二、设计方案：按轮批量判定

### 2.1 设计动机（从服务器压力推导）

逐发roll vs 按轮判定的算力对比：

| 场景 | 逐发roll | 按轮判定 | 节省 |
|---|---|---|---|
| N=8单靶(40轮) | 640次随机调用 | 40次 | 93.75% |
| 大编队混战(100轮) | ~20000次 | 100次 | **99.5%** |

**对大编队混战（设计者最关心的场景），按轮判定省99%算力。** 这就是"N≥2出现固定种子现象"的根源——不是炫技，是省钱。

### 2.2 核心算法

```
战斗初始化：
  seed = deriveSeed(攻击方配置, 防守方配置)
  rng = PRNG(seed)
  模式 = (初始攻击方数量 >= 2) ? 按轮批量 : 逐发roll

每个开火周期（一轮）：
  totalShots = 该轮全舰队总发数（所有存活舰×各自武器×弹药数）
  
  if 模式 == 按轮批量:
    r = rng.next()           // ★ 每轮只消耗1个随机数
    H = binomialInverse(totalShots, p, r)  // 由r反推该轮命中数(整数)
    该轮伤害 = H × perHit
  else: // N=1 逐发roll
    for each 发:
      r = rng.next() 或 Math.random()
      hit = r < p
      ...

  结算：H发命中按确定性优先级分配给同排目标（§1.2）
  击毁判定：target.hp -= 该轮伤害
  
  // ★ 减员处理：下一轮totalShots自然减少，无需特殊处理
```

### 2.3 关键设计决策

**① 每轮1次随机调用（核心）**

逐发roll每发消耗1个随机数；按轮判定每轮只消耗1个。
- 算力从 O(总发数) 降到 O(总轮数)
- 种子推进慢→可复现性自然出现（C1）
- 这就是"固定种子现象"的本质

**② binomialInverse：由均匀随机数反推命中数**

```
输入：总发数M、命中率p、均匀随机数r∈[0,1)
输出：该轮命中数H（整数，0~M）

实现：累积二项分布概率
  累积P(k) = Σ_{i=0}^{k} C(M,i) × p^i × (1-p)^(M-i)
  H = min{ k : 累积P(k) >= r }

特点：
  - H是整数 → 离散扣血（C3✓）
  - E[H] = M×p → 命中率收敛（C5✓）
  - 不同r给出不同H → 每轮波动（C4✓）
  - 同seed同r序列 → 可复现（C1✓）
```

**③ 减员自然处理（解决v1痛点）**

```
第k轮(N=4)：roll → H₁发命中
某舰被击毁 → N变3
第k+1轮(N=3)：totalShots减少 → roll → H₂发命中
```
减员只影响"下一轮开火数变少"，**每轮独立判定，无序列错位**（C7✓）。
这彻底解决了全局序列方案的"减员错位"荒谬。

**④ N=1 与 N≥2 的分界（C2）**

```typescript
模式 = (初始攻击方数量 >= 2) ? 按轮批量 : 逐发roll
```
- N≥2：按轮批量（省算力，确定性，可复现）
- N=1：逐发roll（单舰战斗算力本就小，无需优化；保留真随机）

## 三、约束满足性验证

| 约束 | 如何满足 | 验证 |
|---|---|---|
| C1 可复现 | 每轮1次roll，同seed→同r序列→同H序列 | 9场逐元素相同 ✓ |
| C2 N=1随机 | N=1走逐发roll分支 | 每场不同 ✓ |
| C3 离散 | H是整数，H×perHit是整数倍 | 结构值一格一格掉 ✓ |
| C4 波动 | 不同r→不同H | 每轮命中数不同 ✓ |
| C5 收敛 | E[H]=M×p | 比例≈0.616 ✓ |
| C6 N无关 | 总开火=ΣM≈394/p，N约掉 | N=8/4/2都≈640 ✓ |
| **C7 减员隔离** | **每轮独立roll，减员只改下轮参数** | **无序列错位** ✓ |

**C7是本方案相对v1（全局序列）的决定性优势。**

## 四、伪代码（可直接实现）

```typescript
/**
 * 按轮批量命中判定
 * @param totalShots  该轮全舰队总发数
 * @param finalHitRate 最终命中率（加法公式算出）
 * @param rng         全局共享RNG（每轮消耗1个）
 * @returns 该轮命中数（整数）
 */
function rollHitsPerRound(
  totalShots: number,
  finalHitRate: number,
  rng: BattleRNG
): number {
  const r = rng.next();
  return binomialInverse(totalShots, finalHitRate, r);
}

/**
 * 二项分布逆变换：由均匀r反推命中数
 * 累积二项概率，找第一个使累积概率>=r的k
 */
function binomialInverse(M: number, p: number, r: number): number {
  let cumProb = 0;
  for (let k = 0; k <= M; k++) {
    cumProb += binomialPMF(M, k, p);  // C(M,k)*p^k*(1-p)^(M-k)
    if (cumProb >= r) return k;
  }
  return M;
}

/**
 * 单轮结算（按轮批量模式）
 */
function resolveRound(
  attackers: Ship[],      // 存活的攻击方
  weapon: WeaponSystem,   // 武器（假设同型，简化）
  target: Ship,
  finalHitRate: number,
  rng: BattleRNG
): number {
  const totalShots = attackers.length * weapon.shotsPerCycle;
  const hits = rollHitsPerRound(totalShots, finalHitRate, rng);
  // H发命中的伤害（含暴击：暴击另消耗rng或按比例）
  const perHit = calcPerHit(weapon, target);
  const damage = hits * perHit;
  target.structure -= damage;
  return damage;
}
```

## 七、设计合理性论证

| 设计选择 | 为什么策划会这么选 |
|---|---|
| 按轮而非逐发 | 省99%算力（大编队场景），服务器压力是首要考虑 |
| 离散命中数 | 经得起玩家scrutiny，不会有573.4这种连续分数 |
| 每轮1次roll | 最省算力，顺带产生可复现性（确定性是副产品） |
| N≥2/N=1分界 | N=1算力本就小，无需优化；N≥2才值得切换批量模式 |
| binomialInverse | 把"命中M发中的H发"用1个随机数精确表达，比逐发roll省M倍 |

## ★★★ 关键缺口：NVN复现性未测（2026-07-01）★★★

**现状**：所有"确定性"铁证都来自 **NV1（单靶）**场景。NVN（多对多）的复现性**从未测试**。

**为什么这是决定性缺口**：

| 场景 | 目标选择 | 复现性 | 状态 |
|---|---|---|---|
| NV1（单靶） | 无（唯一靶） | ✅完全可复现 | 已测(铁证) |
| 2V2 同排 | 有，同排确定性优先级 | ✅完全可复现 | 已测(§1.2) |
| 2V2 跨排 | 有，跨排转火 | ❌不可复现 | 已测(§1.2) |
| **NVN（一般混战）** | **有，复杂** | **❓未知** | **未测** |

**三种可能场景**：

| 场景 | NVN表现 | 对v2方案的影响 |
|---|---|---|
| 1. NVN也完全可复现 | 目标选择也确定 | v2直接适用，加目标分配逻辑即可 |
| 2. 部分可复现 | 命中确定，目标分配随机 | v2命中部分对，目标分配需另建 |
| 3. 完全不可复现 | NVN整体真随机 | **v2只适用NV1，"服务器优化"假设错误** |

**场景3的颠覆性含义**：
如果NVN不可复现，则"N≥2固定种子现象"可能**不是服务器优化**，而是NV1单靶场景的特例（单靶无目标选择→天然无随机→看起来确定）。这与"服务器压力下的妥协"假设矛盾。

**已知线索（偏向场景2）**：
- 2V2同排可复现 → 命中判定是确定的（支持v2命中部分）
- 2V2跨排不可复现 → 目标分配有随机（v2的目标分配需另建）
- → 偏向场景2：**命中判定确定，跨排目标分配随机**

**v2方案的适用范围（诚实标注）**：
- ✅ NV1：完全适用（已验证）
- ⚠️ NVN同排：命中判定适用，目标按确定性优先级分配（2V2同排已验证）
- ❓ NVN跨排：命中判定可能适用，但目标分配有随机，复现性未知
- ❓ NVN混舰种混排：完全未知

**建议的验证实验**（优先级最高）：
用同配置重跑一场 **NVN（如4V4或8V8混编）**，观察：
1. 总伤害是否每场相同？（判场景1 vs 2/3）
2. 每艘舰的伤害分布是否相同？（判目标分配是否确定）
3. 击毁顺序是否相同？（判目标优先级是否确定）

**这个实验的结果将决定v2方案的地基是否成立。在测之前，v2应视为"NV1专用近似模型"。**

## ★★★ 决定性答案：NVN不可复现（2026-07-01 策划确认）★★★

**策划确认**：PvE中经常用相同配置打相同的双方配置，**每个战报结果都不同**。

→ **场景3确认：NVN不可复现。**

### 这推翻了"服务器优化"假设

如果"按轮批量判定"是服务器压力下的优化，它应该在NVN（船最多、最该优化）时生效。但**恰恰NVN不生效**（每场不同）。

→ "N≥2固定种子现象"**不是服务器优化**，不是设计者刻意切换的批量模式。

### 正确的图景：确定性来自"无目标选择"，不是"切换模式"

| 场景 | 目标选择 | 复现性 | 根因 |
|---|---|---|---|
| NV1（单靶） | 平凡（唯一靶，没得选） | ✅逐元素相同 | 命中判定确定 + 无目标选择随机 |
| 2V2同排 | 有，但同排确定性优先级 | ✅可复现 | 命中判定确定 + 同排分配确定 |
| NVN（PvE） | 有，含跨排转火 | ❌每场不同 | 目标选择有真随机，掩盖了命中判定的确定性 |

**关键洞察**：
- 命中判定**一直**是确定性的（某种固定规则）——NV1和2V2同排都证实
- 但NV1可复现的**真正原因**是"单靶无目标选择"，不是"切换了批量模式"
- NVN不可复现是因为**目标选择引入真随机**（跨排转火分叉），不是命中判定变随机了

### 对v2方案的最终裁决

| v2假设 | 裁决 | 依据 |
|---|---|---|
| 命中判定按武器批量(确定) | ⚠️ 可能对，但待定具体形式 | NV1/2V2同排支持命中判定确定 |
| 这是服务器优化的结果 | ❌ **错** | NVN不生效→不是优化驱动 |
| 用确定性解释"固定种子现象" | ⚠️ 部分对 | 确定性是真的，但来源是"无目标选择"非"切换模式" |

### 真正的未解核心问题

既然命中判定在NV1和NVN都是确定的（NVN只是被目标选择随机掩盖），那么：

**"命中判定的确定性规则"到底是什么？**

这才是核心未解问题。NV1是研究它的干净场景（无目标选择干扰）。已排除：
- ❌ 连续期望值perHit×p（伤害离散推翻）
- ❌ 统一固定种子逐发roll（解释不了N=1随机，且NVN若也用此应可复现但实测不可）

待研究：
- ⚠️ 命中判定的具体确定性形式（v2的"按武器批量binomial"是候选之一，但动机不是服务器优化）

### 引擎实现建议（修正）

1. **NV1命中判定**：按v2的按武器批量binomial可实现（数据支持），但动机应记为"未知确定性规则"，不是"服务器优化"
2. **NVN目标选择**：需另建目标分配模型，已知跨排转火有真随机（§1.2）
3. **不要假设NVN整体可复现**：实测每场不同，引擎应支持NVN的真随机目标分配
